## 10. バージョン方針のメモ（ver2 → ver2.5 → 拡張版）

- ver2:
  - コアロジック（SGF / GameStore / GoEngine / History）が安定した状態。
- ver2.5（＝シンプル完成版の直前形）:
  - 本メモに沿って
    - 不要機能の削減（Keyboard 操作など）
    - 設定項目の整理と追加
    - UIボタンの責務のシンプル化
  を行う。
  - ver2.5 の状態を、**「シンプル完成版」としてそのままゴールにしてよい**。
- それ以降（拡張版）:
  - 画像認識、問題集管理、高度なレビュー機能などは、
    - 別ブランチ or 別プロジェクト扱いとし、
    - 現在のシンプル完成版とは切り離して検討する。

#### 設計判断の優先順位

- 複雑な例外処理より「常に一貫したシンプルな挙動」を優先する。
- 大多数が迷わないことを基準とし、細かい好み（警告、履歴件数など）は設定側で対応する。
- “後から変えても壊れない部分” はとりあえずの仕様でもよいが、  
  “後で変えると根本から崩れる部分” は先に固める。
- 趣味プロジェクトのため、作者自身が使って心地よい動作を最優先とする。


## 11. ver2 スコープ再整理（2025-11-27 更新）

Phase0 完了後の状況と、デバイス別レイアウト・将来構想を踏まえて、ver2 におけるスコープを再整理する。

### 11.1 ver2 で必ず対応する範囲（Must）

#### 11.1.1 Phase0 一式（済）

- 設定パネル（Basic / Advanced）
- deviceProfile（PC / Tablet / Phone）
- Solve overlay（解答用オーバーレイ）
- Undo / 履歴の整理
- スマホ縦画面でのツールバー高さ調整

上記は既に `main` ブランチに統合済みであり、ver2 の土台として扱う。

#### 11.1.2 スマホ縦レイアウトの確定

- Phone 縦向き（`device-phone` かつ縦）のツールバー構成を ver2 で確定させる。
- 方針（別メモ：`スマホ縦レイアウトの設計前提（Phase1〜ver2系想定）.txt`）に基づき、以下を実装まで行う：

  - 下段 3×4 レイアウトの固定
    - 1行目: `黒配置 / 白配置 / 1手戻る / 1手進む`
    - 2行目: `問題図 / 履歴 / ファイル / 盤面保存`
    - 3行目: `9/13/19 トグル / 機能 / 空白 / 空白`
  - 9/13/19 トグルボタンの実装
    - 単一ボタンで「9 → 13 → 19 → 9 → …」と循環
    - 現行の 9/13/19 切り替えロジックを内部的にラップして呼び出す
  - 上段 2×3 枠のレイアウト
    - 「2×3 という枠数」を確定し、スマホ縦における配置ルールを定める
    - 配置する具体ボタンはツールバー再編の仕様に合わせて決定する

#### 11.1.3 盤面保存機能の不具合修正（番号ズレ）

- 盤面保存の番号表示が内部状態とズレる問題を ver2 までに修正する。
- 原則として UI 表示番号は「現在の並び順に対する 1, 2, 3, …」に統一する。
  - 必要なら内部 ID（`id`）と表示番号を分離し、表示側は配列インデックス + 1 の単純なルールで管理する。
- 修正が困難な場合でも、「番号表示がユーザーの期待を裏切る状態」は残さない。

#### 11.1.4 SGF ヘッダ（軽量メタ情報）の編集

- SGF のごく基本的なヘッダ情報を編集できる UI を用意する。
- 対象とするフィールド（目安）：
  - 対局者名（Black / White）→ `PB` / `PW`
  - 結果 → `RE`
  - 日付 → `DT`
  - 必要に応じてタイトル → `GN` or `EV`
- 取り扱い方針：
  - original SGF はできる限り保持しつつ、「問題用メタ情報」として上書き値を別管理する。
  - SGF 書き出し時は「カスタム値があればそれを優先、なければ original の値を使う」という優先順位とする。

#### 11.1.5 全デバイスで「致命的に壊れていない」状態の保証

- Phone / Tablet / PC の各 deviceProfile で、以下を満たすことを ver2 の条件とする：
  - 盤面・ツールバー・設定パネルがレイアウトとして成立し、操作不能レベルの崩れがない。
  - 小さな「見た目の野暮ったさ」は許容するが、
    - ボタンが画面外に完全に隠れる、
    - 盤が極端に細長くなり実質的に読めない、
    といった状態は ver2 では残さない。


### 11.2 ver2.5 以降に回す範囲（Nice-to-have）

以下は「現時点でニーズは認識しているが、ver2 完了の必須条件とはしない」ものとする。

#### 11.2.1 タブレット（特に 4:3 比率）の見た目最適化

- iPad mini など 4:3 画面比率での以下の調整：
  - 盤サイズとツールバーのバランス最適化
  - ボタンの行数・列数・表示密度の微調整
- ver2 では「普通に使える（致命的に崩れていない）」ことを優先し、
  見た目の最適化は ver2.5 以降のタスクとする。

#### 11.2.2 スマホ横画面の“快適さ”向上

- Phone 横向きでのレイアウト：
  - 盤を左右どちらかに寄せ、反対側をボタンカラムとするシンプルな構成を基本とする。
  - 横 3〜4 ボタン × 縦並び程度のレイアウトを想定。
- ver2 の段階では「どうにか使える（盤が見えて、ボタンが押せる）」レベルを達成目標とし、
  その上の「快適さ」の追求は ver2.5 以降とする。

#### 11.2.3 タブレット・PC のツールバー美観調整

- 縦横・解像度・画面比率ごとの細かなボタン数や行数の最適化。
- 「見た目として美しいバランス」「情報量の整理」など、UX 的な仕上げ。
- これらは ver2 のコアスコープではなく、ver2 完了後に落ち着いて調整する。


### 11.3 ver2 では扱わない範囲（Out-of-scope）

以下は ver2 のスコープ外とし、別フェーズ（例：PC解析モード、ver3 以降）で検討する。

#### 11.3.1 PC 解析モード・外部エンジン連携

- PC ローカルでの本格的な解析モード。
- KataGo 等の外部エンジンとの連携機能（ローカル実行を含む）。
- これらはアプリ構造や GoEngine 設計を大きく変える可能性が高いため、
  ver2 では「構想メモの整理」にとどめ、実装は別プロジェクトとして扱う。

#### 11.3.2 問題集 JSON ＋ 半自動生成

- 問題集 JSON フォーマットの確立と、その半自動生成（エンジン解析等を含む）。
- PC 解析モードやエンジン連携と密接に関係するため、
  ver2 のブラウザ版スコープからは切り離して考える。

#### 11.3.3 画像認識系機能（例：野狐の盤面読み取り）

- スクリーンショット等から盤面を自動認識する機能（例：野狐囲碁の盤面読み取り）。
- 重い処理になること、および non-goals としていた「画像認識系拡張」に該当するため、
  ver2 では実装しない。
- 将来の PC 向け拡張や別ツールとして検討する。

#### 11.3.4 大規模なキーボードショートカット設計

- PC 向けの網羅的なショートカット体系の設計・実装。
- 現時点では優先度を下げ、必要最低限のショートカットにとどめる。
- 本格的なショートカット整理は、PC UI の大幅見直しとセットで検討する。

## 2025-xx-xx KeyboardController 削除

- 旧バージョンで使用していた `KeyboardController` を削除した。
- 盤面ショートカット機能は今後も提供しない方針とし、必要になった場合のみ別設計で検討する。
