## 2. ブランチ運用に関するルール

ブランチ運用は原則として `docs/10-branch-strategy.md` を正とする。  
Codex / DeepResearch は、次の点を絶対に変更提案してはならない。

- `stable-20251116` の役割
  - 最新安定版を指す公開用ブランチ
  - GitHub Pages の公開元ブランチ
  - 通常開発では直接コミット禁止

- GitHub Pages の公開設定
  - 公開元は **`stable-20251116 / (root)`** とする
  - `dist/` を公開ルートに変更する提案を行わない

- `dist/` の扱い（`docs/11` および `docs/12` に従う）
  - リリース時に `main` 上で `npm run build` して更新する
  - 開発ブランチ（`feature/*`, `codex/*`）では `dist/` を触らない
  - `stable-20251116` 上で `dist/` を直接編集しない

## 3. 修正レベルと対応方法

作業の規模・影響度は `.claude/rules/01-correction-levels.md` で定義される「Lv0〜5」に従う。

概要（詳細は `.claude/rules/01-correction-levels.md` を参照）:

| Lv | 規模 | 対応方法 |
|:--:|------|----------|
| 0 | 超軽微（文言/定数） | Claude Code で直接修正 |
| 1 | 軽微（〜50行、1ファイル） | Claude Code で直接修正 |
| 2 | 中程度（1ファイル内で機能変更） | Plan Mode → 承認 → 実行 |
| 3 | 複数ファイル（2-3ファイル） | Plan Mode + 段階的実行 |
| 4 | 大規模（3ファイル超/構造変更） | 分割案を先に提示 |
| 5 | 根本変更 | 設計相談のみ（実装保留） |

Claude Code は修正前に必ずレベルを判定し、適切なフローを選択する。

## 4. Claude Code に任せてよい範囲

Claude Code に任せてよい典型例:

- コード品質改善（デバッグコード削除、コメント整理）
- 広範囲にまたがるレイアウト調整やリファクタリング（複数ファイル）
- 状態管理・ヒストリ管理などの大きめな内部構造の整理
- ドキュメントの整理・更新
- `ver3` 以降の「内部基盤整備」系タスクのうち、仕様が明確なもの

次のようなものは、原則として **任せない**:

- ブランチ戦略・リリースフロー・`dist/` の扱いを変える提案
- GitHub Pages の設定を変更する提案
- リポジトリ構成を大きく変える提案（フォルダ構成の全面変更など）
- non-goals に明記されている機能（重い AI 連携の本格組込みなど）
- Git操作の直接実行（commit/push/PR作成）→ ユーザーに指示のみ

## 5. コード品質管理

### 5.1 デバッグコードの管理

**本番コードに含めないもの**:
- `console.log` による状態ダンプ
- デバッグ用の一時的な関数（例: `logBoardDump()`）
- テスト用のコメントアウトコード

**保持するもの**:
- `console.error` によるエラーハンドリング
- 実際のユーザーへのエラーメッセージ
- 重要な警告ログ

### 5.2 技術的負債の分類（T1-T3）

| 分類 | 説明 | 対応優先度 |
|:----:|------|:----------:|
| T1 | バグの温床（今後のバグを引き起こす） | 高 |
| T2 | 将来機能の足かせ | 中 |
| T3 | 気持ち悪さ（動作には影響なし） | 低 |

## 7. この文書の更新ポリシー

- ブランチ戦略やリリースフローに変更が入った場合は、必ず本ファイルも更新する
- `docs/` 内の他ファイルと内容が矛盾した場合、本ファイルを優先せず、
  先に該当トピックの「本体ドキュメント」（10, 11, 12 など）を更新して整合を取る
- Claude Code の利用方針が変わった場合は、人間の開発者がレビューした上で追記する
