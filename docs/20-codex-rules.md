# 20 – Codex / DeepResearch 利用ルール

この文書は、Codex や DeepResearch などの「コード自動生成系 AI」が  
`sentoku870/tumego` リポジトリで作業するときの前提・禁止事項・参照順序を定義する。

人間の開発者も「AI に何を任せるか」を確認するために参照してよい。

---

## 1. 参照するドキュメントの優先順位

同じトピックについて複数の資料が存在する場合、**次の優先順位**で解釈し、  
下位の資料が上位と矛盾している場合は「上位を正」とする。

1. `docs/` 配下の公式ドキュメント  
   - 例:  
     - `docs/00-overview.md`  
     - `docs/10-branch-strategy.md`  
     - `docs/11-release-flow.md`  
     - `docs/12-dist-handling.md`  
     - `docs/20-codex-rules.md`（この文書）

2. リポジトリ直下の「サブプロンプト系ガイド」
   - `GitHub_Operation_Sub-Prompt.md`  
   - `Correction_Level_Definition_Sub-Prompt.md`  
   - `Codex_Request_Instruction_Sub-Prompt.md`  
   - `UI_Design_Guideline_Sub-Prompt.md`  

3. その他の設計メモ・テキスト資料
   - 例: `アーキテクチャ概要.md`, `プロジェクト分析.md`, `ver2 分析レポート.md` など

Codex / DeepResearch は、まず 1 → 2 → 3 の順に確認し、  
矛盾があれば 1 を優先し、独自ルールを追加しない。

---

## 2. ブランチ運用に関するルール

ブランチ運用は原則として `docs/10-branch-strategy.md` を正とする。  
Codex / DeepResearch は、次の点を絶対に変更提案してはならない。

- `stable-20251116` の役割
  - 最新安定版を指す公開用ブランチ
  - GitHub Pages の公開元ブランチ
  - 通常開発では直接コミット禁止

- GitHub Pages の公開設定
  - 公開元は **`stable-20251116 / (root)`** とする
  - `dist/` を公開ルートに変更する提案を行わない

- `dist/` の扱い（`docs/11` および `docs/12` に従う）
  - リリース時に `main` 上で `npm run build` して更新する
  - 開発ブランチ（`feature/*`, `codex/*`）では `dist/` を触らない
  - `stable-20251116` 上で `dist/` を直接編集しない

## 3. 複雑度レベルと誰が対応するか

作業の規模・影響度は、  
`Correction_Level_Definition_Sub-Prompt.md` で定義される「レベル0〜5」に従う。

概要だけここに再掲する（詳細は元ファイルを参照する）:

- **レベル0: 超軽微**
  - 数行の修正、位置が明確、コアロジックに影響しない
  - 対応: GPT で直接修正し、簡易コミット

- **レベル1: 軽微**
  - 150行以内、依存関係が少ない追加や変更
  - 対応: GPT で直接修正、簡易コミット

- **レベル2: 中程度単一ファイル**
  - 1ファイル完結、700行以下、振る舞い変更を含む
  - 対応: GPT で修正、PR 標準フロー

- **レベル3: 複数ファイル**
  - 2〜3ファイル、各300行以下
  - 対応: GPT で修正、PR 標準フロー

- **レベル4: 大規模**
  - 3ファイル超 or 700行超
  - 対応: Codex（`codex/*` ブランチ）に依頼、PR 標準フロー

- **レベル5: 根本変更**
  - 設計見直しが必要
  - 対応: まず機能妥協や段階分割を検討し、安易に実装しない

Codex / DeepResearch を使う場合は、**主にレベル4以上** を対象とし、  
レベル0〜3は可能な範囲で GPT でのスポット対応を優先する。

---

## 4. Codex / DeepResearch に任せてよい範囲

Codex / DeepResearch に任せてよい典型例:

- 広範囲にまたがるレイアウト調整やリファクタリング（複数ファイル）
- 状態管理・ヒストリ管理などの大きめな内部構造の整理
- テストコードの一括生成（既存仕様に沿うもの）
- `ver3` 以降の「内部基盤整備」系タスクのうち、仕様が明確なもの

次のようなものは、原則として **任せない**:

- ブランチ戦略・リリースフロー・`dist/` の扱いを変える提案
- GitHub Pages の設定を変更する提案
- リポジトリ構成を大きく変える提案（フォルダ構成の全面変更など）
- non-goals に明記されている機能（重い AI 連携の本格組込みなど）

## 5. Codex 依頼時の基本フロー（人間側の準備）

人間の開発者が Codex に大きめの作業を依頼する場合は、  
次のような情報をまとめてプロンプトとして渡す（詳細テンプレは  
`Codex_Request_Instruction_Sub-Prompt.md` を参照）。

1. **目的**  
   - 何を達成したいか（例: 「解答モードの Undo/Redo ロジックを整理したい」）

2. **対象範囲**  
   - 関連ファイル名と、触ってよい範囲／触ってはいけない範囲

3. **依存する仕様ドキュメント**
   - どの docs ファイルを正としてほしいか  
   - 例: `docs/00-overview.md`, `docs/10-branch-strategy.md`, `docs/11-release-flow.md` など

4. **複雑度レベルの目安**
   - レベル4以上が想定される場合に Codex を使う

5. **完了条件**
   - どうなっていれば「完了」とみなすか（動作・コードの観点両方）

## 6. 関連ガイドファイル一覧

AI は次のファイルを「自分向けのサブプロンプト」として扱う。  
人間の開発者にとっても、AI がどのように動くべきかを知る参考資料となる。

- `GitHub_Operation_Sub-Prompt.md`  
  - ブランチ運用と Git 操作フロー（簡易コミット / PR 標準 など）

- `Correction_Level_Definition_Sub-Prompt.md`  
  - レベル0〜5の定義と、どのレベルで誰がどう対応するか

- `Codex_Request_Instruction_Sub-Prompt.md`  
  - Codex への依頼文を書くときの具体的なテンプレ・アンチパターン

- `UI_Design_Guideline_Sub-Prompt.md`  
  - UI 実装時のスタイル・コンポーネント利用方針

これらのファイル名は、AI 向けであることを明示するために  
`*_Sub-Prompt.md` という名前を付けている。  
**Codex / DeepResearch は、これらを通常の Markdown と同様に読み込める前提とする。**

---

## 7. この文書の更新ポリシー

- ブランチ戦略やリリースフローに変更が入った場合は、必ず本ファイルも更新する
- `docs/` 内の他ファイルと内容が矛盾した場合、本ファイルを優先せず、  
  先に該当トピックの「本体ドキュメント」（10, 11, 12 など）を更新して整合を取る
- Codex / DeepResearch の利用方針が変わった場合は、人間の開発者がレビューした上で追記する
