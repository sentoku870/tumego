```markdown
# 12 – dist/ ディレクトリの扱い

## 1. dist/ の役割

- `dist/` は、`npm run build` で生成される **ビルド成果物** を保存するディレクトリ
- GitHub Pages は `stable-20251116` ブランチ上の `dist/` から公開される

## 2. 基本ルール

1. **Git 管理対象とする**
   - `.gitignore` からは除外し、`dist/` はリポジトリで管理する
   - ただし、更新はリリース時のみに限定する

2. **日常開発では触らない**
   - 通常の機能開発やバグ修正では `dist/` を変更しない
   - PR では原則 `dist/` を含めない（含まれていたら差し戻し対象）

3. **リリース時のみ更新**
   - `11-release-flow.md` に記載の手順に従い、`npm run build` の直後に `dist/` をコミットする
   - `dist/` コミットと stable ブランチ同期は **セット** で行う

4. **コンフリクト対応**
   - もし `dist/` でコンフリクトが起きた場合、
     - 原則として **再ビルドして上書き** する
     - 手で中身をマージしない（ソースから再生成できるため）

## 3. よくある NG パターン

- `feature/*` や `codex/*` ブランチで `dist/` を含めてしまう
- テストが通っていない状態で `dist/` を更新し、そのまま stable に反映する
- `stable-20251116` 上で直接 `dist/` を編集する

いずれもリリースの整合性を壊す可能性が高いため禁止する。
````

```markdown
# 20 – Codex / DeepResearch 利用ルール

この文書は、Codex や DeepResearch などの「コード自動生成系 AI」を
`sentoku870/tumego` リポジトリで利用する際のルールをまとめたものです。

## 1. 参照ドキュメントの優先順位

Codex / GPT / DeepResearch などの AI には、次の順番でドキュメントを参照させる:

1. `docs/` 配下（このフォルダ）のドキュメント  
   - **最優先の公式仕様**
2. リポジトリ直下の各種ガイド
   - `GitHub操作ガイド.md`
   - `修正レベル詳細.md`
   - `Codex依頼ガイド.md`
   - `PR問題対応ガイド.md`
   - `UI設計ガイド.md`
   - `ヒントモード＋AnalysisEngine v1` など将来機能の仕様書
3. その他のテキスト資料（アーキテクチャ概要 / プロジェクト分析 / non-goals など）

docs/ と他の資料の内容が矛盾する場合、**docs/ の内容を正とする。**

## 2. Codex に任せてよい作業

- 複数ファイルにまたがるリファクタリングや大きめの UI 変更
- TypeScript / CSS / HTML の構造的な修正
- 既存のガイドライン（UI 設計ガイド、修正レベル、non-goals など）に沿った機能追加

ただし、必ず以下を守る:

- 作業用ブランチは `codex/...` を使う
- PR を作成し、人間側がレビューしてから `main` にマージする

## 3. Codex に禁止する作業

Codex / DeepResearch に対しては、次の操作を **禁止** する:

- `stable-20251116` ブランチの変更（直接コミット・force push など）
- GitHub Pages の設定変更（公開ブランチやルートディレクトリの変更）
- `dist/` ディレクトリの更新や削除
  - `dist/` を変更する必要があるリリース作業は、必ず人間が実施する
- ブランチ保護やリポジトリ設定の変更

## 4. 修正レベルと Codex 依頼

- 修正の規模に応じて、`修正レベル詳細.md` のレベル 0〜5 を目安にする
- レベル 4 以上（3 ファイル超の大規模変更・設計見直しなど）の場合は、
  `Codex依頼ガイド.md` に沿って Codex に依頼するか、
  「機能妥協」を検討する
- PR に問題が出た場合は、`PR問題対応ガイド.md` のフローに従って原因特定と修正を行う

## 5. 将来の仕様書の置き場

- 新しい機能（例: ヒントモード＋AnalysisEngine v1、GoEngine 設計見直しなど）の仕様書は、
  原則としてリポジトリ直下または `テキスト資料/` などに md/txt で追加する
- 重要なルール部分（UI 原則・禁止事項・運用ルールなど）は、
  必要に応じて `docs/` に短くサマリを追記し、
  「docs が最終的な公式ルール」となるように保つ
```
