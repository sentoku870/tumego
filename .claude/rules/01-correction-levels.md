# 修正レベル定義（Correction Levels）- Web版

> このファイルは修正規模の判定基準を定義します。
> Claude Code は修正前に必ずレベルを判定し、適切なフローを選択してください。

---

## 1. レベル一覧

| Lv | 名称 | 規模イメージ | 対応方法 |
|:--:|------|-------------|----------|
| 0 | 超軽微 | 文言/定数だけ（挙動ほぼ不変） | 直接修正 |
| 1 | 軽微 | 短いファイル内の小さな修正/スタイル調整 | 直接修正 |
| 2 | 中程度 | 1ファイル内で機能をしっかり変える/追加 | Plan Mode → 承認 |
| 3 | 複数ファイル | 2-3ファイル、UI＋ロジックなど | Plan Mode + 段階的 |
| 4 | 大規模 | 3ファイル超、構造/状態管理レベル変更 | 分割案を先に提示 |
| 5 | 根本変更 | 設計全体の見直しが必要 | 設計相談のみ |

※迷ったら上のレベルに寄せる（安全側）。

## 2. docs-only / code-change の境界

### docs-only（ドキュメントのみ）
- `docs/**/*.md`
- README、CHANGELOG など

→ **main直接コミット可**

### code-change（コード変更を含む）
- `src/**/*.ts`, `*.js`, `*.css`, `*.html`
- `package.json`, 設定ファイル
- `dist/` への影響がある変更

→ **feature ブランチ + PR 必須**

### 迷ったら
**code-change（安全側）** として扱う。

## 3. レベル別の詳細

### Lv0: 超軽微
- **典型例**: 表示文言修正、CSS色調整、定数変更
- **回答で必ず書くこと**:
  - 変更ファイル名
  - 変更内容（1-2行）
  - ブラウザでの確認ポイント

### Lv1: 軽微
- **典型例**: 1ファイル内の小さなバグ修正、スタイル微調整
- **回答で必ず書くこと**:
  - 変更ファイル名
  - 具体的な差分
  - 動作確認ポイント

### Lv2: 中程度
- **典型例**: 1ファイル内で機能追加、イベントハンドラ変更
- **回答で必ず書くこと**:
  - 「Lv2」宣言
  - 影響範囲（どの機能/画面が変わるか）
  - PC/スマホ両方での確認ポイント

### Lv3: 複数ファイル
- **典型例**: UI＋ロジック、状態管理＋表示の連動
- **回答で必ず書くこと**:
  - 「Lv3」宣言
  - どのファイルを触るか
  - 既存機能への影響

### Lv4: 大規模
- **典型例**: GameStore の構造変更、モード切替ロジック刷新
- **回答で必ず書くこと**:
  - 変更の段階分割（フェーズ提案）
  - 「今回やる範囲 / やらない範囲」の線引き

### Lv5: 根本変更
- **典型例**: 状態管理の抜本再設計、SGF処理の全面書き換え
- **回答で必ず書くこと**:
  - 「Lv5」宣言
  - 目的と非目的
  - 分割案（Lv2-3に切り出せる単位）

## 4. tumego Web版でよくある判断例

| 変更内容 | 判定 | 理由 |
|----------|------|------|
| CSSの色・サイズ調整 | Lv0-1 | 見た目のみ |
| ボタンのテキスト変更 | Lv0 | 挙動不変 |
| イベントハンドラのロジック修正 | Lv1-2 | 1ファイル内 |
| GameStore + BoardView の連動修正 | Lv3 | 複数ファイル |
| モード切替（edit/solve/view）の仕様変更 | Lv3-4 | 広範囲に影響 |
| SGF パーサーの全面刷新 | Lv4-5 | 構造変更 |

## 5. ver3 デバッグ一時停止中の注意

現在 ver3 デバッグは一時停止中（PC版優先）。

再開時の方針：
- まず現状の問題点を整理
- Lv ごとに優先順位付け
- T1（バグの温床）→ T2（将来機能の足かせ）→ T3（気持ち悪さ）の順で対応
